    <div class="post-text" itemprop="text">

<p>I have been using python selenium for web automation testing. The key part of automation is to find the right element for a user-visible object in a HTML page.  The following API will work most of the time, but not all the time.</p>

<pre class="lang-py prettyprint prettyprinted" style=""><code><span class="pln">find_element_by_xxx</span><span class="pun">,</span><span class="pln">  xxx can be id</span><span class="pun">,</span><span class="pln"> name</span><span class="pun">,</span><span class="pln"> xpath</span><span class="pun">,</span><span class="pln"> tag_name etc</span><span class="pun">.</span><span class="pln"> </span></code></pre>

<p>When HTML page is too complicated, I would like to search the dom tree. Wonder if it's possible to ask the selenium server to serialize the entire DOM  (with the element id that can be used to perform action on through webdriver server).  Client side (python script) can do its own search algorithm to find the right element. </p>

<p>Note that python selenium can get the entire html page by </p>

<pre class="lang-py prettyprint prettyprinted" style=""><code><span class="pln">drv</span><span class="pun">.</span><span class="pln">page_source</span></code></pre>

<p>However, parsing this doesn't give the internal element id from  selenium server's point of view, hence not useful.</p>

<p><strong>EDIT1:</strong>
    Paraphrase it to make it more clear (thanks @alecxe):  what's needed here is a serialized representation of all the DOM elements (with their DOM structure preserved) in the selenium server,  this serialized representation can be sent to the client side (a python selenium test app) which can do its own search.</p>

<div class="answercell post-layout--right">
    
    <div class="post-text" itemprop="text">
<h3>The Problem</h3>

<p>Ok, so there may be cases where you need to perform some substantial processing of a page on the client (Python) side rather than on the server (browser) side. For instance, if you have some sort of machine learning system already written in Python and it needs to analyze the whole page before performing actions on them, then although it is possible to do it with a bunch of <code>find_element</code> calls, this gets very expensive because each call is a round-trip between the client and the server. And rewriting it to work in the browser may be too expensive.</p>

<h3>Why Selenium's Identifiers wont' do it</h3>

<p>However, I do not see an <em>efficient</em> way to get a serialization of the DOM <em>together</em> with Selenium's own identifiers. Selenium creates these identifiers on an as-needed basis, when you call <code>find_element</code> or when DOM nodes are returned from an <code>execute_script</code> call (or passed to the callback that <code>execute_async_script</code> gives to the script). But if you call <code>find_element</code> to get identifiers for each element, then you are back to square one. I could imagine decorating the DOM in the browser with the required information but there is no public API to request some sort of pre-assignment of <code>WebElement</code> ids. As a matter of fact, these identifiers are designed to be opaque so even if a solution managed somehow to get the required information, I'd be concerned about cross-browser viability and ongoing support.</p>

<h3>A Solution</h3>

<p>There is however a way to get an addressing system that would work on both sides: XPath. The idea is to parse the DOM serialization into a tree on the client side and then get the XPath of the nodes you are interested in and use this to get the corresponding WebElement. So if you'd have to perform dozens of client-server roundtrips to determine which single element you need to perform a click on, you'd be able so reduce this to an initial query of the page source plus a single <code>find_element</code> call with the XPath you need.</p>

<p>Here is a super simple proof of concept. It fetches the main input field of the Google front page.</p>

<pre class="lang-py prettyprint prettyprinted" style=""><code><span class="kwd">from</span><span class="pln"> </span><span class="typ">StringIO</span><span class="pln"> </span><span class="kwd">import</span><span class="pln"> </span><span class="typ">StringIO</span><span class="pln">

</span><span class="kwd">from</span><span class="pln"> selenium </span><span class="kwd">import</span><span class="pln"> webdriver
</span><span class="kwd">import</span><span class="pln"> lxml</span><span class="pun">.</span><span class="pln">etree

</span><span class="com">#</span><span class="pln">
</span><span class="com"># Make sure that your chromedriver is in your PATH, and use the following line...</span><span class="pln">
</span><span class="com">#</span><span class="pln">
driver </span><span class="pun">=</span><span class="pln"> webdriver</span><span class="pun">.</span><span class="typ">Chrome</span><span class="pun">()</span><span class="pln">
</span><span class="com">#</span><span class="pln">
</span><span class="com"># ... or, you can put the path inside the call like this:</span><span class="pln">
</span><span class="com"># driver = webdriver.Chrome("/path/to/chromedriver")</span><span class="pln">
</span><span class="com">#</span><span class="pln">

parser </span><span class="pun">=</span><span class="pln"> lxml</span><span class="pun">.</span><span class="pln">etree</span><span class="pun">.</span><span class="typ">HTMLParser</span><span class="pun">()</span><span class="pln">

driver</span><span class="pun">.</span><span class="pln">get</span><span class="pun">(</span><span class="str">"http://google.com"</span><span class="pun">)</span><span class="pln">

</span><span class="com"># We get this element only for the sake of illustration, for the tests later.</span><span class="pln">
input_from_find </span><span class="pun">=</span><span class="pln"> driver</span><span class="pun">.</span><span class="pln">find_element_by_id</span><span class="pun">(</span><span class="str">"gbqfq"</span><span class="pun">)</span><span class="pln">
input_from_find</span><span class="pun">.</span><span class="pln">send_keys</span><span class="pun">(</span><span class="str">"foo"</span><span class="pun">)</span><span class="pln">

html </span><span class="pun">=</span><span class="pln"> driver</span><span class="pun">.</span><span class="pln">execute_script</span><span class="pun">(</span><span class="str">"return document.documentElement.outerHTML"</span><span class="pun">)</span><span class="pln">
tree </span><span class="pun">=</span><span class="pln"> lxml</span><span class="pun">.</span><span class="pln">etree</span><span class="pun">.</span><span class="pln">parse</span><span class="pun">(</span><span class="typ">StringIO</span><span class="pun">(</span><span class="pln">html</span><span class="pun">),</span><span class="pln"> parser</span><span class="pun">)</span><span class="pln">

</span><span class="com"># Find our element in the tree.</span><span class="pln">
field </span><span class="pun">=</span><span class="pln"> tree</span><span class="pun">.</span><span class="pln">find</span><span class="pun">(</span><span class="str">"//*[@id='gbqfq']"</span><span class="pun">)</span><span class="pln">
</span><span class="com"># Get the XPath that will uniquely select it.</span><span class="pln">
path </span><span class="pun">=</span><span class="pln"> tree</span><span class="pun">.</span><span class="pln">getpath</span><span class="pun">(</span><span class="pln">field</span><span class="pun">)</span><span class="pln">

</span><span class="com"># Use the XPath to get the element from the browser.</span><span class="pln">
input_from_xpath </span><span class="pun">=</span><span class="pln"> driver</span><span class="pun">.</span><span class="pln">find_element_by_xpath</span><span class="pun">(</span><span class="pln">path</span><span class="pun">)</span><span class="pln">

</span><span class="kwd">print</span><span class="pln"> </span><span class="str">"Equal?"</span><span class="pun">,</span><span class="pln"> input_from_xpath </span><span class="pun">==</span><span class="pln"> input_from_find
</span><span class="com"># In JavaScript we would not call ``getAttribute`` but Selenium treats</span><span class="pln">
</span><span class="com"># a query on the ``value`` attribute as special, so this works.</span><span class="pln">
</span><span class="kwd">print</span><span class="pln"> </span><span class="str">"Value:"</span><span class="pun">,</span><span class="pln"> input_from_xpath</span><span class="pun">.</span><span class="pln">get_attribute</span><span class="pun">(</span><span class="str">"value"</span><span class="pun">)</span><span class="pln">

driver</span><span class="pun">.</span><span class="pln">quit</span><span class="pun">()</span></code></pre>

<p>Notes: </p>

<ol>
<li><p>The code above does not use <code>driver.page_source</code> because Selenium's documentation states that there is no guarantee as to the freshness of what it returns. It could be the state of the current DOM or the state of the DOM when the page was first loaded.</p></li>
<li><p>This solution suffers from the exact same problems that <code>find_element</code> suffers from regarding dynamic contents. If the DOM changes while the analysis is occurring, then you are working on a stale representation of the DOM.</p></li>
<li><p>If you have to generate JavaScript events while performing the analysis, and these events change the DOM, then you'd need fetch the DOM again. (This is similar to the previous point but a solution that uses <code>find_element</code> calls could conceivably avoid the problem I'm talking about in <em>this</em> point by ordering the sequence of calls carefully.)</p></li>
<li><p><code>lxml</code>'s tree could possibly differ <em>structurally</em> from the DOM tree in such a way that the XPath obtained from <code>lxml</code> does not address the corresponding element in the DOM. What <code>lxml</code> processes is the cleaned up serialized view that the browser has of the HTML passed to it. Therefore, <strong>so long as the code is written to prevent the problems I've mentioned in point 2 and 3</strong>, I do not see this as a likely scenario, but it is not impossible.</p></li>
</ol>

<p>See my <a href="https://stackoverflow.com/a/25646401/1906307">other answer</a> for the issues regarding any attempts at getting Selenium's identifiers.</p>

<p>Again, the problem is to reduce a bunch of <code>find_element</code> calls so as to avoid the round-trips associated with them.</p>

<p>A different method from my other answer is to use <code>execute_script</code> to <strong>perform the search on the browser</strong> and then return all the elements needed. For instance, this code would require three round-trips but can be reduced to just one round-trip:</p>

<pre class="lang-py prettyprint prettyprinted" style=""><code><span class="pln">el</span><span class="pun">,</span><span class="pln"> parent</span><span class="pun">,</span><span class="pln"> text </span><span class="pun">=</span><span class="pln"> driver</span><span class="pun">.</span><span class="pln">execute_script</span><span class="pun">(</span><span class="str">"""
var el = document.querySelector(arguments[0]);
return [el, el.parentNode, el.textContent];
"""</span><span class="pun">,</span><span class="pln"> selector</span><span class="pun">)</span></code></pre>

<p>This returns an element, the element's parent and the element's textual contents on the basis of whatever CSS selector I wish to pass. In a case where the page has jQuery loaded, I could use jQuery to perform the search. And the logic can get as complicated as needed.</p>

<p>This method takes care of the vast majority of cases where reducing round-trips is desirable but it does not take care of a scenario like the one I've given in illustration in my other answer.</p>
